# CLAUDE.md

このファイルはClaude Codeがこのリポジトリで作業する際のルールを定義します。

## 重要: 工数・AI活用記録のヒアリング

**コミットまたはPR作成を依頼された場合、必ずユーザーに以下をヒアリングしてください:**

1. **実作業時間** - ユーザーが実際に作業した時間
2. **AI使用時間** - AIを使用した時間
3. **API使用料** - わかる場合（不明な場合は$0.00）
4. **AI未使用時の想定作業時間** - AIを使わなかった場合の想定時間
5. **AI活用度** - 低/中/高/非常に高

### ヒアリング例

```
コミット/PR作成のため、工数・AI活用記録を教えてください:
- 実作業時間:
- AI使用時間:
- API使用料:
- AI未使用想定時間:
- AI活用度（低/中/高/非常に高）:
```

ユーザーが情報を提供しない場合や省略を希望する場合は、推定値を使用してコミット/PRを作成してください。

## コミットルール

### フォーマット

```
<Type>: <簡潔な説明>

<詳細な説明（任意）>

AI: <AIサービス> | 時間: <使用時間> | 費用: <API使用料>
実作業: <実作業時間> | AI未使用想定: <想定時間>
活用度: <低/中/高/非常に高>

Co-Authored-By: Claude <noreply@anthropic.com>
```

### Type一覧

| Type | 説明 |
|------|------|
| Add | 新機能追加 |
| Fix | バグ修正 |
| Update | 機能改善 |
| Remove | 機能削除 |
| Docs | ドキュメント |
| Refactor | リファクタリング |

### ルール

- 1行目は50文字以内推奨（72文字上限）
- 本文は72文字で折り返し
- 命令形で記述（"Added"ではなく"Add"）
- 工数・AI活用記録は必ず記載する
- Co-Authored-Byを記載する

### AI活用度の目安

| レベル | 範囲 | 説明 |
|--------|------|------|
| 低 | 0-25% | 主に手作業、AIは補助的 |
| 中 | 26-50% | AIと手作業を併用 |
| 高 | 51-75% | AIが主導、人間がレビュー・修正 |
| 非常に高 | 76-100% | ほぼAIが実施、人間は指示・確認のみ |

### コミット例

```
Add: SSH接続のリトライ機能

接続失敗時に最大3回までリトライする機能を追加。
指数バックオフでリトライ間隔を調整。

AI: Claude Code | 時間: 0h 45m | 費用: $0.30
実作業: 1h 00m | AI未使用想定: 3h 00m
活用度: 高

Co-Authored-By: Claude <noreply@anthropic.com>
```

## PR作成ルール

### PRテンプレートの必須項目

1. **概要** - 変更内容の説明
2. **変更内容** - 具体的な変更点をリスト形式で
3. **動機**（任意） - なぜこの変更が必要か
4. **テスト方法** - 動作確認の手順
5. **工数・AI活用記録** - 以下を必ず記入
   - 使用したAIサービスと時間
   - 実作業時間
   - AI未使用時の想定作業時間
   - AI活用度（チェックボックスで選択）

### チェックリスト

PRを作成する前に以下を確認:

#### 基本チェック
- [ ] コードスタイルに準拠している
- [ ] テストを実施した
- [ ] ドキュメントを更新した（必要な場合）
- [ ] 工数・AI活用記録を記入した

#### 設計・実装整合性チェック（main向けPR必須）
- [ ] 設計ドキュメントに記載された機能は全て実装されている
- [ ] 設計にない実装が存在しない
- [ ] 機能の対応／未対応ステータスが正確に記載されている

#### 進捗ドキュメントチェック（進捗ドキュメント更新時）
- [ ] 進捗ステータスが実態と一致している
- [ ] 完了タスクには対応する実装が存在する
- [ ] 未着手タスクには実装が存在しない

### 自動チェック

PRを作成すると以下のワークフローが自動実行されます:

| ワークフロー | 対象 | 内容 |
|-------------|------|------|
| PR Effort & AI Usage Check | 全PR | 工数・AI活用記録の記入チェック |
| Design-Implementation Consistency | main向けPR | 設計と実装の整合性チェック |
| Progress Document Consistency | 進捗ドキュメント含むPR | 進捗ステータスの正確性チェック |

### PR作成コマンド例

```bash
gh pr create --base main --head develop --title "Add: 機能名" --body "$(cat <<'EOF'
## Summary

- 変更点1
- 変更点2

## Test plan

- [ ] テスト項目1
- [ ] テスト項目2

🤖 Generated with [Claude Code](https://claude.com/claude-code)
EOF
)"
```

## ブランチ戦略

- `main`: 本番環境用、設計と実装が完全に一致している状態
- `develop`: 開発用、機能開発はここから分岐
- `feature/*`: 機能開発用
- `fix/*`: バグ修正用
- `docs/*`: ドキュメント更新用
- `refactor/*`: リファクタリング用
